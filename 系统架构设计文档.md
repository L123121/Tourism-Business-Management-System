# 旅游业务管理系统 — 系统架构设计文档

## 一、系统概述

本系统是一个面向旅行社的业务管理系统，采用 B/S（浏览器/服务器）架构，实现旅游路线管理、旅游活动管理、旅游团管理、旅游申请办理、参加者信息管理、余款支付、财务数据导出等核心业务功能。系统通过角色权限控制，为不同岗位的工作人员提供差异化的功能界面。

### 2.3 数据库

| 技术 | 说明 |
|------|------|
| SQLite | 轻量级嵌入式关系型数据库，无需单独部署，适合中小型系统 |

### 2.4 架构模式

```
┌─────────────┐     HTTP/REST     ┌─────────────┐     SQL      ┌──────────┐
│   浏览器     │ ◄──────────────► │  Flask 后端  │ ◄───────────► │ SQLite   │
│  Vue 3 SPA  │    JSON 数据      │  Python API │   读写数据    │ 数据库   │
└─────────────┘                   └─────────────┘              └──────────┘
```

- **前端**：单页应用（SPA），通过 Vue Router 实现客户端路由，通过 Axios 调用后端 RESTful API
- **后端**：Flask 提供 RESTful API，使用 JWT Token 进行身份认证，通过装饰器实现角色权限控制
- **数据库**：SQLite 单文件数据库，通过 Python 内置 `sqlite3` 模块访问

## 三、系统角色与权限设计

### 3.1 角色定义

系统定义了 5 种角色：

| 角色标识 | 角色名称 | 职责说明 |
|----------|----------|----------|
| `receptionist` | 前台员工 | 负责办理旅游申请、管理参加者信息、处理申请变更/取消 |
| `collector` | 催款员工 | 负责余款催收、打印确认书和交款单 |
| `routeAdmin` | 路线管理员 | 负责旅游路线、旅游活动、旅游团、价格的管理与发布 |
| `accountant` | 会计人员 | 负责财务数据查询与导出 |
| `admin` | 系统管理员 | 拥有全部权限，可管理所有功能模块和用户账号 |

| 菜单 | 功能 | 说明 |

### 3.3 后端 API 权限控制

系统通过两层权限控制保障安全：

**第一层：登录认证（JWT Token）**
- 用户登录后服务器生成 JWT Token，有效期 8 小时
- 前端在每次请求的 `Authorization` 头中携带 Token
- 后端通过 `@login_required` 装饰器验证 Token 有效性

**第二层：角色权限校验（API 路径匹配）**
- 后端通过 `@role_required` 装饰器校验用户角色
- 每个角色定义了可访问的 API 路径前缀列表
- `admin` 角色拥有全部 API 的访问权限

各角色 API 权限映射：

| 角色 | 可访问 API 路径前缀 |
|------|---------------------|
| receptionist | `/api/applications`, `/api/participants`, `/api/groups`, `/api/routes`, `/api/activities`, `/api/prices`, `/api/calc-`, `/api/stats` |
| collector | `/api/balance`, `/api/applications`, `/api/groups`, `/api/routes`, `/api/activities`, `/api/stats` |
| routeAdmin | `/api/routes`, `/api/activities`, `/api/groups`, `/api/prices`, `/api/stats` |
| accountant | `/api/finance`, `/api/applications`(只读), `/api/stats` |
| admin | `/api/*`（全部） |

### 3.4 前端权限控制

前端根据用户角色动态控制页面访问和数据展示：

**路由守卫**：通过 `router.beforeEach` 检查用户菜单权限，无权限时跳转首页并提示。

**首页数据权限**：根据角色动态调用后端 API：
| 角色 | 可查看首页数据 |
|------|----------------|
| 所有角色 | 统计数据（本月申请数、今日新增等） |
| receptionist / collector / accountant / admin | 最近申请记录 |
| collector / admin | 余款即将到期列表 |

**API 错误处理**：响应拦截器支持 `suppressError` 配置，首页等场景下静默处理 403 错误，避免弹出"权限不足"提示。

### 5.2 核心业务规则

| 业务规则 | 说明 |
|----------|------|
| 订金计算 | 距出发 ≥60天：300元/人；≥30天：600元/人；≥15天：1000元/人；<15天：600元/人 |
| 取消费率 | 距出发 ≥30天：10%；≥15天：30%；≥7天：50%；<7天：100% |
| 人数限制 | 申请参加人数 + 已报名人数 ≤ 团队容量 |
| 价格发布 | 价格发布后不可再修改，确保已售出订单价格一致 |
| 余款付清 | 支付余款后累计金额 ≥ 总费用时，申请状态自动变为"已完成" |
| 余款逾期取消 | 仅取消未出发且余款截止日期已过的申请；已出发未付清的申请保持"进行中"，留给收费员催款 |

### 5.3 启动时自动处理

后端启动时自动执行以下任务：

| 任务 | 说明 |
|------|------|
| 数据库初始化 | 创建表结构（如不存在） |
| 种子数据填充 | 初始化路线、活动、旅游团、申请等业务数据 |
| 余款逾期检查 | 自动取消未出发且余款截止日期已过的申请 |

## 六、系统启动与运行

### 6.1 后端启动

```bash
cd backend
pip install -r requirements.txt
python run.py
```

后端运行在 `http://localhost:5000`，首次启动自动初始化数据库、填充种子数据、检查逾期申请。

### 6.2 前端启动

```bash
cd frontend
npm install
npm run dev
```

前端开发服务器运行在 `http://localhost:5173`，通过 Vite 代理将 `/api` 请求转发到后端。

### 6.3 生产部署

前端执行 `npm run build` 构建后，将 `dist/` 目录内容复制到 `backend/static/`，Flask 同时承担静态文件服务和 API 服务，单一端口 `5000` 即可运行完整系统。

## 七、单元测试

### 7.1 测试框架与运行方式

| 项目 | 说明 |
|------|------|
| 测试框架 | Python `unittest` |
| 运行命令 | `cd backend && python run_tests.py` |
| 覆盖率报告 | `python run_tests.py --coverage`（需安装 `coverage`） |

### 7.2 测试基类（`tests/__init__.py`）

`TourismTestCase` 提供以下测试基础设施：

- 每个测试类使用独立的临时 SQLite 数据库文件，测试之间互不干扰
